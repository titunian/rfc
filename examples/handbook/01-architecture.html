Architecture

How orfc actually fits together — a Next.js web app, a Postgres store, a TypeScript CLI, and a single sanitization choke-point that keeps user-authored HTML safe to render.

Figure 1 · Request flows. CLI authenticates with an API key; browsers use a session cookie. Both hit the same Next.js API.

Stack

Frontend
Next.js 14 · React 18 · Tailwind
App Router, server components for the public read path, client components for the editor and comment UI.
Database
Postgres · Drizzle ORM
Neon serverless Postgres in production, JSON-file fallback in local dev so the CLI stays demo-able offline.
CLI
Node 18+ · Commander · TypeScript
Single binary at @orfc/cli. Reads ~/.orfc/config.json for credentials.
Auth & mail
NextAuth · Resend · Slack webhooks
Email-OTP login for both web and CLI. Optional Slack webhook for review notifications.
Render path

A plan with contentType: "html" skips the react-markdown pipeline entirely. The body goes through DOMPurify with the html + svg + svgFilters profiles, then drops into a typography-free .orfc-html-doc container. The author's CSS variables and inline styles are preserved verbatim — orfc never imposes its own theme on an HTML doc.

Invariants

  • 1 Every write goes through /api/plans. The CLI is a thin client.
  • 2 Updates always snapshot the previous version into plan_versions — nothing is overwritten in place.
  • 3 Sanitization happens at render, not at write. Stored content is the author's exact bytes; what the browser sees is filtered.
  • 4 Comment anchors are DOM-based (block index + text snippet), so they survive the markdown ↔ HTML render-mode change.
