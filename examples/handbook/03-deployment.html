Deployment

How orfc gets from git push to orfc.dev. Vercel handles build + edge, Neon handles Postgres, schema migrations are still a manual step (until they're not).

Step 3 is the trip-wire. The 2026-05-10 outage skipped it; the deploy went green while the schema lagged behind by two columns.

Environment variables

Key Required Used by
DATABASE_URL prod Neon connection string. Empty in local dev → JSON file fallback.
NEXTAUTH_SECRET all Session signing.
NEXTAUTH_URL prod Callback URL base. https://orfc.dev in prod.
RESEND_API_KEY prod OTP email and review notifications.
APP_URL opt Override for the URL embedded in API responses. Defaults to request origin.
Trip-wire

Schema-touching PRs need db push before merge, not after. Vercel auto-deploys main on merge, so a code-and-schema PR that skips the db step ends up deployed against an old schema and the API blows up everywhere. Until the migration is wired into the deploy pipeline, run it first.

Running a migration

From the repo root, with prod DATABASE_URL in your shell or .env.production.local: 

# option A — drizzle migrate (tracks applied migrations)
npm run db:migrate

# option B — drizzle push (diffs schema.ts vs DB, no journal)
npm run db:push

# option C — vercel-managed env
vercel env pull .env.production.local
npx -y dotenv-cli -e .env.production.local -- npx drizzle-kit migrate

push is fine for solo / small-team work — no migration files needed. Switch to migrate once you have multiple writers and want a paper trail.
